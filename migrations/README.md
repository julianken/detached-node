This directory contains Payload CMS database migrations for the Detached Node project.

## ⚠️ Current Status: Development Setup

**Important**: This application has been running in Payload dev mode, which auto-pushes schema changes directly to the database without using migrations. The migration files exist but have NOT been applied to the development database.

**For Production Deployment**: Migrations will run automatically during the first production build. Start with a fresh database.

## Running Migrations

### Development (Current State)

**Migrations are NOT currently applied to the dev database.** The database schema exists from running Payload in dev mode (auto-push).

To apply migrations to dev database (warning: will require database reset):

```bash
# Option 1: Reset and migrate (DESTROYS ALL DATA)
npm run db:reset  # Resets database and re-seeds test data

# Option 2: Manually mark migrations as applied (preserves data)
# Connect to database and run:
INSERT INTO payload_migrations (name, batch, updated_at, created_at) VALUES
  ('20260213_071339_add_composite_indexes', 1, NOW(), NOW()),
  ('20260213_072000_add_composite_indexes', 1, NOW(), NOW());
```

### Production (First Deployment)

Migrations run automatically during first production build:

```bash
npm run build:production
# Runs: payload migrate && next build
```

**Requirements**:
1. Start with empty Postgres database
2. Set `DATABASE_URL` environment variable
3. Run build command - migrations will apply automatically
4. Migrations are tracked in `payload_migrations` table

**For Subsequent Deployments**:
Only NEW migrations (created after production launch) will run automatically.

## Validation

After running migrations, validate with:

```bash
# Verify indexes created
psql $DATABASE_URL -f scripts/validate-indexes.sql

# Check migration status
npm run payload migrate:status
```

## Troubleshooting

### Migration Fails

```bash
# Check database connection
echo $DATABASE_URL
psql $DATABASE_URL -c "SELECT version();"

# Check migration status
npm run payload migrate:status

# View migration logs
npm run payload migrate -- --verbose
```

### Rollback Failed Migration

```bash
# Automatic rollback
npm run payload migrate:rollback

# Manual rollback (if needed)
psql $DATABASE_URL << 'EOF'
DROP INDEX IF EXISTS idx_pages_status_updatedAt;
DROP INDEX IF EXISTS idx_posts_status_updatedAt;
DROP INDEX IF EXISTS idx_posts_featured_status_publishedAt;
DROP INDEX IF EXISTS idx_posts_status_publishedAt;
EOF
```

## Best Practices

1. **Always backup** before running migrations in production
2. **Test migrations** in development first
3. **Validate results** with `validate-indexes.sql` script
4. **Monitor performance** after deployment
5. **Document custom migrations** with inline comments
